Todo Dialog と Redux のデータフロー

ダイアログと一覧は同じ Redux Store を参照します。入力欄からフォーカスを外すと、API 保存を待たずに Store を更新するため、背面の一覧も即座に再描画されます。

フォーカス離脱時の流れ

1. 入力ダイアログで Todo 名または詳細を編集します。
2. bluronBlur がドラフトの確定処理を呼びます。
3. ReduxtodoChanged が対象の Todo を置換します。
4. 再描画一覧とダイアログが同じ Store の値で更新されます。

ダイアログ側

TodoDialog は入力値をローカル state として保持し、フォーカスを外した時点で Redux に渡します。

function commitDraft() {
  if (todo) {
    onDraftChange({ ...todo, title, description, completed })
  }
}

<input
  value={title}
  onChange={(event) => setTitle(event.target.value)}
  onBlur={commitDraft}
/>

編集中の入力体験を保つために local state を使い、他の画面へ共有する値だけを Redux に反映します。

Redux Slice

todoChanged は ID が一致する Todo のみを差し替えます。ほかの Todo やリクエスト状態には影響しません。

todoChanged(state, action: PayloadAction<Todo>) {
  const index = state.todos.findIndex(
    (todo) => todo.id === action.payload.id,
  )

  if (index !== -1) {
    state.todos[index] = action.payload
  }
}

Hook の責務

useTodos が UI と Redux を接続します。ダイアログは Repository や API 実装を知らずに済みます。

const updateDraft = useCallback(
  (todo: Todo) => {
    dispatch(todoActions.todoChanged(todo))
  },
  [dispatch],
)
  • todoChanged: blur 時の即時表示更新。API 呼び出しは行いません。
  • update: 保存ボタンでユースケースを実行し、API へ永続化します。
  • requestSucceeded: API から再取得した正しい一覧で Store を確定します。

設計上のポイント

一覧とダイアログが別々の state を持つと表示のずれが起こります。共有データである Todo は Redux に一元化し、表示専用の入力途中値だけをダイアログの local state に残します。
